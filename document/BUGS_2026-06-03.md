## BUG #1 — `_decode_params` F5 sigmoid saturation (CRITICO)

### Posizione
`core/network.py:380-381`

```python
raw_eq = raw / self.decode_scale         # decode_scale = [1.0, 0.054, 0.108, 0.0594, 0.0676]
return self.param_lo + (self.param_hi - self.param_lo) * torch.sigmoid(raw_eq)
```

### Descrizione
F5 pre-scaling è stato disegnato per "equalizzare il gradiente" matematicamente, MA produce un effetto collaterale grave: amplifica `raw` di 9-18× per i params T/s0/a/b, portando `raw_eq` fuori dalla zona lineare di sigmoid. Sigmoid satura → derivative → 0 → gradient → 0 → params bloccati al bound dal random init.

### Evidenza empirica (random init, no training)

| Param | A1 baseline | A8 attn | Decode amplification |
|---|---:|---:|---:|
| v0 | 0% saturated | 0% | 1.0× |
| T | **97% saturated** | 46% | 18.5× |
| s0 | **96% saturated** | 28% | 9.2× |
| a | 5% | 90% | 16.8× |
| b | 36% | 92% | 14.8× |

### Conseguenza
Backward gradient su LI weights per canale (A1 baseline, random init):
- s0: `0.00004` (5000× più piccolo di a)
- b: `0.00006` (5000× più piccolo di a)
- T: 0.099 (basso)
- v0: 0.191
- a: 0.370

**s0 e b non possono imparare**.

### Fix proposto
Rimuovere F5 amplification. Usare semplicemente:
```python
return self.param_lo + (self.param_hi - self.param_lo) * torch.sigmoid(raw)
```

Il problema originale (v0 impara più veloce di T) si risolve meglio con scheduler/wd, non con sigmoid amplification.

## BUG #2 — Xavier asymmetric init bias su LI output (GRAVE)

### Posizione
`core/network.py:63` (OutputLayer_LI init):
```python
self.fc_weight = nn.Parameter(torch.Tensor(out_features, in_features))
nn.init.xavier_uniform_(self.fc_weight)
```

### Descrizione
Xavier_uniform produce pesi distribuiti uniformemente in `[-bound, +bound]`. Per ogni riga (output channel), la mean dei pesi è **NON zero** (varianza casuale). 

Quando l'input al LI sono spike binari {0, 1}, il current = Σ weight × spike ha bias atteso = `weight.row_mean × spike_rate × hidden_size`. Per spike_rate=0.05 e hidden=32 → bias ≈ row_mean × 1.6.

Esempio (seed=42, A1 baseline):
- out_T row_mean = +0.037 → bias positivo → raw_T positivo → sigmoid HIGH → T = 2.5 (max)
- out_s0 row_mean = -0.027 → bias negativo → raw_s0 negativo → sigmoid LOW → s0 = 1.0 (min)
- out_a row_mean = +0.077 → bias forte positivo → a inclina HIGH

Combinato con BUG #1, **determina deterministicamente QUALE bound viene saturato** per ogni canale, dal random init.

### Fix proposto
Sottrarre la mean per riga post-Xavier:
```python
nn.init.xavier_uniform_(self.fc_weight)
with torch.no_grad():
    self.fc_weight.sub_(self.fc_weight.mean(dim=1, keepdim=True))  # bias-zero per row
```

Oppure aggiungere bias learnable inizializzato a 0:
```python
self.bias = nn.Parameter(torch.zeros(out_features))
# poi in forward: current = linear(spikes, fc_weight) + self.bias
```

## CRITICITÀ #3 — ALIF cascata signal-dying (A2/A4)

### Posizione
`core/network.py:626-695` (CF_FSNN_Net_Stacked).

### Descrizione
Stacked di N HiddenLayer_ALIF in serie: layer 0 produce pochi spike (1-2% al n_ticks=10), layer 1+ riceve quei pochi spike → current piccolo → potential non raggiunge threshold (base_th=1.5) → 0 spike → layer 2 riceve 0 → 0 spike → raw_out = 0.

Empirico: A2 e A4 producono **raw_out = 0.0000 SEMPRE** al random init.

### Fix proposto
Per ALIF cascata, abbassare base_threshold per layer non-input:
```python
# In CF_FSNN_Net_Stacked.__init__:
layers = []
in_dim = self.CF_INPUT_SIZE
for i in range(n_hidden):
    layer = HiddenLayer_ALIF(in_dim, hidden_sizes[i], ...)
    if i > 0:
        with torch.no_grad():
            layer.cell.base_threshold.fill_(1.0)  # invece di 1.5 default
    layers.append(layer)
```

Oppure usare A3 skip-connection pattern (= già funziona).

## CRITICITÀ #4 — Delay mask penalty in HiddenLayer_ALIF (MEDIO)

### Posizione
`core/network.py:48-51`:
```python
current = torch.zeros(...)
for d in range(self.max_delay):
    mask_d = (self.delays == d).float()
    current += linear(self.x_buffer[d], w_po2 * mask_d)
```

### Descrizione
`self.delays` è (out_features, in_features) di int random in [0, max_delay). Per ogni edge (i,j), il delay assegnato è random. La mask `(delays == d)` è binaria → solo gli edge con `delays==d` contribuiscono.

In media, ogni edge contribuisce 1/max_delay del tempo. Quindi il current effettivo è ~1/max_delay del current senza delay → ALIF necessita più tick per raggiungere threshold.

NB: questo è BY DESIGN (delayed synapses per FPGA) ma rende la propagazione più lenta. È **causa contributiva** di criticità #3 (A2/A4 dead output).

### Fix proposto
Non urgente. Se necessario, ridurre max_delay (es. 6 → 3) o usare init pesi più grande per compensare.
